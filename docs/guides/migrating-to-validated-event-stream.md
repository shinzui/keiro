# Migrating to `ValidatedEventStream`

This guide is for **downstream consumers of keiro** (services and apps that build
`EventStream`s and run commands against them) migrating across the change
introduced by keiro ExecPlan
[`docs/plans/84-enforce-replay-safety-with-a-validatedeventstream-at-the-keiro-command-boundary.md`](../plans/84-enforce-replay-safety-with-a-validatedeventstream-at-the-keiro-command-boundary.md).

It is written to be followed with no prior context beyond your own repository.
The registered downstream projects at the time this guide was written are
`danwa`, `kanmon`, `kawa`, `keiro-runtime-docs`, `keiro-runtime-jitsurei`,
`keiro-runtime-patterns`, `kikan`, `kioku`, `kizashi`, `kotei`, `mori-app`, and
`shikigami`; `mori-app` also has a package-level dependency on keiro.

For the conceptual guarantee behind the migration, see
[Replayability Safety](../user/replay-safety.md).


## What changed, in one paragraph

keiro runs **event-sourced aggregates**: it replays a stream of stored events
through a pure state machine (a keiki *transducer*) to rebuild the aggregate's
current state before each new command. An aggregate is **replay-safe** only if
every command field a transition reads is recoverable from the events it emits;
a transition that changes state from information it never puts on the wire is a
**hidden input**, and it makes the aggregate silently diverge on replay. keiro
could already *detect* this (`validateTransducer`) but never *forced* the check.
Now it does: the command runners accept only a `ValidatedEventStream`, a newtype
that only the validation smart constructors can produce (by running the
replay-safety, determinism, and dead-edge checks). A bare `EventStream` record
can no longer reach a runner — it fails to type-check.


## The good news first

- **The migration is compiler-driven.** The change is a *type* change, so when you
  bump keiro your build fails with exact errors at every site that constructs an
  `EventStream` and hands it to a runner or a Router / ProcessManager / Projection
  config. There is no silent behavior change to hunt for.
- **There is no data migration.** `ValidatedEventStream` is a `newtype`, erased at
  runtime. No event payload, snapshot, or stream format changes. No downtime, no
  backfill, no coordinated cutover. Each consumer migrates independently, whenever
  it bumps keiro.
- **If your service builds and its aggregates were already sound, the migration is
  mechanical.** The only way it is *not* mechanical is if you have a latent
  replay-safety bug — see "The one thing that can block you" below. That is the
  guard doing its job.


## Prerequisites

1. A keiro release (git pin or version) that contains ExecPlan 84.
2. If you use keiro-dsl to generate aggregates: a matching keiro-dsl release whose
   scaffolder emits validated streams (ExecPlan 84, Milestone M4). Bump both
   together — your generated code and the runners must agree.


## Which path are you on?

**Path A — DSL-generated aggregates.** Your `EventStream` modules are checked-in,
`@generated by keiro-dsl` files (typically `…/Generated/EventStream.hs`), produced
from a `.keiro` spec. You do **not** hand-edit them.

**Path B — hand-authored aggregates.** You wrote the `EventStream` record literal
yourself.

A single service can be on both paths for different aggregates. Do each aggregate
under its own path.


## Path A — DSL-generated: regenerate, do not hand-edit

The generator now emits, for each aggregate `Foo`, a bare record under
`fooEventStreamDef :: FooEventStreamDef` plus a validated binding under the
original `fooEventStream :: FooEventStream` name. The generated
`FooEventStreamDef` alias is the raw `EventStream ...`; the generated
`FooEventStream` alias is the `ValidatedEventStream ...`. Because the validated
value keeps the original name, your runner call sites
(`runCommand ... fooEventStream ...`) compile unchanged.

Steps:

1. Bump your keiro **and** keiro-dsl pins to the releases from Prerequisites.
2. **Regenerate** from your `.keiro` spec with the new scaffolder — the same
   command you used when the modules were first generated. Do not hand-edit the
   `@generated` files.
3. Rebuild. The regenerated `fooEventStream :: FooEventStream` (now a
   `ValidatedEventStream`) satisfies the runners; call sites are untouched.

Why regeneration is required, not optional: your old generated
`fooEventStream = EventStream { … }` is a *bare* record. A plain keiro bump without
regeneration leaves that bare record in your tree, and it will not type-check
against the new runners. You must re-run the scaffolder to pick up the validated
binding.

Note on safety: keiro-dsl already generates a harness that asserts
`validateTransducer defaultValidationOptions … == []`. If your service builds and
runs that harness in CI today, your aggregates are already replay-safe, so the
regenerated `mkEventStreamOrThrow` cannot throw and this migration is purely
mechanical.

Note on stream handles: `Stream` phantoms remain tagged with the raw
`EventStream` shape. If you have hand-written annotations such as
`stream "foo-1" :: Stream FooEventStream` against a regenerated DSL alias, change
them to `Stream FooEventStreamDef`.


## Path B — hand-authored: wrap each construction site

Use the `…Def`-rename pattern so your call sites do not churn. For each aggregate:

```haskell
-- before
fooEventStream :: FooEventStream            -- = EventStream (HsPred …) …
fooEventStream = EventStream { … }

-- after
type FooEventStreamDef = FooEventStream     -- or inline the EventStream … type
type ValidatedFooEventStream = ValidatedEventStream (HsPred …) … FooCommand FooEvent

fooEventStreamDef :: FooEventStreamDef      -- the former bare record, renamed
fooEventStreamDef = EventStream { … }       -- body unchanged

fooEventStream :: ValidatedFooEventStream    -- = ValidatedEventStream (HsPred …) …
fooEventStream = mkEventStreamOrThrow "Foo" fooEventStreamDef
```

`mkEventStreamOrThrow :: (HasCallStack, Bounded s, Enum s, Ord s, Show s) => Text ->
EventStream … -> ValidatedEventStream …` is the partial constructor: it throws at
startup if the stream is not replay-safe. It is the right choice when you have a
sibling test (a keiki-style harness) proving the aggregate is sound. If you would
rather handle unsafety explicitly, use the **total** constructor and match the
`Left` at your wiring/startup boundary:

```haskell
fooEventStream :: Either [EventStreamWarning] ValidatedFooEventStream
fooEventStream = mkEventStream "Foo" fooEventStreamDef
-- …resolve the Either where you assemble your runtime, failing fast on Left.
```

Because the validated value keeps the name `fooEventStream`, every
`runCommand ... fooEventStream ...` call site is unchanged. Keep existing
`Stream FooEventStream` annotations on stream handles if `FooEventStream` remains
your raw alias; only the stream *value* passed to runners becomes validated.

Requirement: `mkEventStream` needs `(Bounded s, Enum s, Ord s, Show s)` on your
aggregate's vertex (control-state) type. Derive all four if you have not already
(`deriving stock (Eq, Ord, Show, Enum, Bounded)` on the vertex enum).


## Router, ProcessManager, and Projection config

If you embed an `EventStream` in a config record, its field type changes to
`ValidatedEventStream`:

- **Router** (`Keiro.Router`) — the `targetEventStream` field is now
  `ValidatedEventStream …`. Assign the validated value (`fooEventStream`) to it.
- **ProcessManager** (`Keiro.ProcessManager`) — the `eventStream` and
  `targetEventStream` fields are now `ValidatedEventStream …`. The
  `streamFor :: Text -> Stream (EventStream …)` field is a *stream handle*, not a
  stream value, and is unchanged.
- **Projection** (`Keiro.Projection`) — the EventStream-taking projection runner
  now takes `ValidatedEventStream …`. (If your projections are `InlineProjection`
  values wired into a projection worker, they do not embed an `EventStream` and are
  unaffected.)

In all three, if you need to read a field off the embedded stream (for example
`resolveStreamName`), go through the accessor:
`unvalidated (cfg ^. #targetEventStream) ^. #resolveStreamName`.


## The one thing that can block you: a latent replay-safety bug

If one of your aggregates is *actually* replay-unsafe today, `mkEventStream`
returns `Left` (and `mkEventStreamOrThrow` throws at startup). You cannot wrap your
way past it — that is the entire point. Two resolutions:

1. **It is a real bug** (a hidden input — a transition reads a command field it
   never emits). Fix the aggregate: emit an event carrying that field, or move the
   state derivation onto an emitting edge. On DSL-generated services, fix the
   `.keiro` spec / hole and regenerate.
2. **It is a conservative false positive.** Only the *determinism* and *dead-edge*
   checks can false-positive; the *hidden-input* check does not. If you have
   confirmed the warning is benign, narrow the options for that one stream:

   ```haskell
   fooEventStream = either (error . show) id $
     mkEventStreamWith
       defaultValidationOptions { checkDeterminism = False }  -- or checkReachability
       "Foo"
       fooEventStreamDef
   ```

   Leave a comment explaining why the narrowing is safe. Never narrow
   `failOnEpsilonReadsInput` — that is the replay-safety check this whole change
   exists to enforce.

Because this case can surface real, previously-hidden bugs, do not schedule your
fleet migration as a single find-and-replace across all services at once. Migrate
one service at a time and leave room to fix an aggregate if the checker flags one.


## Verify

1. **It builds.** `cabal build all` (or your project's build) succeeds. Any
   remaining `Couldn't match type 'EventStream …' with 'ValidatedEventStream …'`
   error names an un-migrated construction or wiring site — apply Path A or B
   there.
2. **Commands still work.** Run your existing command/integration tests. Since the
   newtype is transparent at runtime, behavior is unchanged for sound aggregates.
3. **The guard is live.** Optionally, temporarily pass a bare `fooEventStreamDef`
   to a runner and confirm it fails to compile with a `ValidatedEventStream` type
   error; revert.


## Rollback

Revert the keiro/keiro-dsl pin bump and (Path A) regenerate from the old
scaffolder, or (Path B) drop the `…Def` rename. There are no database migrations,
no persisted-format changes, and no external state — the newtype is compile-time
only — so rollback is a pure source-level revert.


## Fleet sequencing (for the maintainer landing this across all consumers)

1. Land ExecPlan 84 in keiro; cut a keiro release.
2. Ship the keiro-dsl scaffolder change (M4); cut a keiro-dsl release.
3. Migrate the worked-example repos first (`keiro-runtime-docs`,
   `keiro-runtime-jitsurei`, `keiro-runtime-patterns`) — they double as the
   reference migration everyone copies.
4. Migrate the service/app consumers one at a time, each on its own schedule. No
   big-bang is needed: there is no runtime protocol change, so a service on the old
   keiro and a service on the new keiro interoperate at the event-store level
   unchanged.
