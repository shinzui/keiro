# Gate snapshot staleness on fold changes

This ExecPlan is a living document. The sections Progress, Surprises & Discoveries,
Decision Log, and Outcomes & Retrospective must be kept up to date as work proceeds.
If durable project context changes, update or create ADRs in docs/adr/ in the same change.


## Purpose / Big Picture

Today a keiro service can change how its aggregate folds events into state — edit one
transition's `update` expression, redeploy — and silently keep serving snapshots that were
computed under the *old* fold. The snapshot lookup discriminates only on
`(state_codec_version, regfile_shape_hash)`, and neither component sees fold logic, the
control-state type, or slot-type internals. The result is the worst gap found by the July
2026 evolution review: a deploy that passes every gate and then makes command decisions from
state that no full replay would ever produce — silently, and (because the post-append replay
witness re-persists the stale-derived state as a *fresh* snapshot at the new head version)
compoundingly.

After this plan, three things are true. First, the snapshot discriminator gains a third
component, a *state shape hash* of the control-state type (computed generically in keiki),
so changing the shape of the folded state invalidates old snapshots automatically. Second,
services generated by `keiro-dsl` get a *fold fingerprint* — a digest of the spec's
transition surface (guards, writes, emits, targets, states, register initials) — composed
into that third component, so any spec-visible fold change invalidates old snapshots
automatically, and `keiro-dsl diff` prints an advisory whenever the transition surface
changes. Third, the manual contract that remains for hand-written services ("bump
`stateCodecVersion` when you change fold logic the discriminator cannot see") is actually
written down — in the `StateCodec` haddocks, in the generated code, and in the one factually
wrong paragraph of `docs/user/snapshots.md` that today tells readers the opposite.

You can see it working by running the new keiro test that reproduces the stale-fold scenario:
before this change a snapshot written under one fold discriminator is served to a codec with
a different fold fingerprint; after this change the lookup misses, hydration falls back to
full replay, and the freshly persisted snapshot carries the new discriminator.

## Decision Log

- Decision: The fold-sensitive discriminator is a new `state_shape_hash` column (and a new
  `stateShapeHash` field on `StateCodec`), not extra input folded into the existing
  `regfile_shape_hash`.
  Rationale: A separate column keeps diagnostics honest — an operator inspecting
  `keiro.keiro_snapshots` can see *which* axis invalidated a seed (register layout vs
  state/fold), and the existing column keeps meaning exactly what its name says. The
  rejected alternative (hashing the state shape into the regfile hash input) avoids a
  migration but silently changes the meaning of a column that `docs/user/snapshots.md`
  documents, and gives the DSL no place to compose its fold fingerprint without further
  overloading. Cost: a `StateCodec` record-field addition is API-breaking for hand-written
  construction sites (accepted; this repo releases its packages in lockstep and the next
  release train is a breaking one).
  Date: 2026-07-23

- Decision: The DSL fold fingerprint is composed into `state_shape_hash` via a public
  keiro combinator (`withFoldFingerprint`), producing a value of the form
  `<keiki-state-shape-hash>;fold=<fingerprint>`, rather than getting a fourth column.
  Rationale: The keiki state-shape hash and the spec fingerprint answer the same question —
  "is this stored seed the product of the current fold?" — so they share the discriminator
  slot. The composed text remains human-readable in the table. Hand-written services that
  want the same rigor can call `withFoldFingerprint` with their own token; those that do not
  keep the documented manual `stateCodecVersion` contract.
  Date: 2026-07-23

- Decision: The fingerprint digest reuses keiro-dsl's existing FNV-1a-64 helper style
  (`Keiro.Dsl.ReadModelShape.deriveShapeHash` precedent) rather than adding a crypto
  dependency.
  Rationale: This is a change detector, not a security boundary. A 64-bit collision has the
  same failure mode as today's behavior (a stale seed accepted) at ~2^-64 likelihood per
  changed spec, and the library already ships the FNV fold with a documented no-bytestring
  rationale (`keiro-dsl/src/Keiro/Dsl/ReadModelShape.hs:33-58`).
  Date: 2026-07-23

- Decision: `verifyAndSnapshot`'s re-persist behavior is kept as-is; no seed-provenance
  gate is added.
  Rationale: The re-persist was only dangerous because a stale seed could be *loaded*. With
  the discriminator fixed, any seed that hydration accepts already matches the current
  codec version, register shape, state shape, and (for DSL services) fold fingerprint, so
  re-persisting derived state is sound. The residual exposure — a hand-written service that
  changes fold logic without bumping `stateCodecVersion` — is exactly the documented manual
  contract, and a provenance gate cannot detect it either (the discriminators are equal by
  construction in that case). Recorded here so plan 141 can quote it.
  Date: 2026-07-23

- Decision: This plan edits only the factually false sentence of
  `docs/user/snapshots.md:80-83`; the full rewrite of that section belongs to plan 141
  (docs/plans/141-correct-the-evolution-documentation-and-deploy-ordering-guidance.md),
  which owns all evolution-doc edits per master plan 24's Integration Points.
  Rationale: Leaving a known-false claim live while the gate ships would misdirect users;
  anything beyond the minimal correction would collide with plan 141's pass.
  Date: 2026-07-23

## Context and Orientation

This repository (`/Users/shinzui/Keikaku/bokuno/keiro`) contains the keiro event-sourcing
runtime as several cabal packages: `keiro-core` (pure contracts: codecs, the `EventStream`
record, validation), `keiro` (the Postgres-backed runtime: command cycle, snapshots,
process managers, workflows), `keiro-dsl` (the `.keiro` spec toolchain: parser, validator,
scaffolder, differ, harness), `keiro-migrations` (the framework's SQL migrations), plus
test-support and example packages. The sibling repository
`/Users/shinzui/Keikaku/bokuno/keiki` contains keiki, the pure symbolic state-machine
library keiro builds on; keiro consumes it as a released package (`keiki >=0.2 && <0.3` in
`keiro/keiro.cabal:115` and `keiro-core/keiro-core.cabal:59`; currently 0.2.0.0).

Architectural ground truth this plan depends on: keiki has no separate decide/evolve
functions — one edge set is both forward stepping and replay. Replay re-inverts each stored
event back to a command and re-checks the edge guard against it
(`keiki/src/Keiki/Core.hs:1223-1228`: `solveOutput` reconstructs the command, `models
(guard e) (regs, ci)` re-evaluates the guard). So editing a guard, output, or update
expression edits the *interpretation of the already-stored log*. keiro surfaces replay
failures as `HydrationNoInvertingEdge` / `HydrationAmbiguousInversion` /
`HydrationQueueMismatch` (`keiro/src/Keiro/Command.hs:457-462`). Upcasters run at
decode-time forever — hydration decodes every stored event through the codec chain on every
replay (`keiro/src/Keiro/Command.hs:410`) — and no stored-data migration mechanism exists
anywhere in the stack.

The snapshot mechanism, precisely. A snapshot stores an encoded `(state, registers)` seed
per stream in `keiro.keiro_snapshots` (created by
`keiro-migrations/migrations/0001-keiro-bootstrap.sql:8-19`; columns `stream_id` (primary
key), `stream_version`, `state` (jsonb), `state_codec_version`, `regfile_shape_hash`,
timestamps). The lookup statement filters on codec version and shape hash
(`keiro/src/Keiro/Snapshot/Schema.hs:107-114`), and the write statement persists both
(`Schema.hs:122-147`). Both values come from the stream's `StateCodec`
(`keiro/src/Keiro/Snapshot.hs:143-151` copies `stateCodecVersion` and `shapeHash` into the
row). `StateCodec` itself is defined in `keiro-core/src/Keiro/EventStream.hs:102-108`
(fields `stateCodecVersion :: !Int`, `shapeHash :: !Text`, `encode`, `decode`).

The default codec, `defaultStateCodec` (`keiro/src/Keiro/Snapshot/Codec.hs:38-48`), takes a
hand-supplied integer version and computes `shapeHash = regFileShapeHash (Proxy @rs)` (line
41) — the *register file only*. The control-state type `s` is not hashed at all.
`regFileShapeHash` (`keiki/src/Keiki/Shape.hs:195-239`) is a single SHA-256 over
`<slotName>:<canonicalTypeName>;` per slot in slot-list order, and `canonicalTypeName`
(`Shape.hs:68-83`) renders the type's *name* (module-qualified by default, pinned names for
built-ins) — a slot type's internal constructors and fields are invisible to it. keiki
exports no transducer fingerprint of any kind. DSL-generated services build their codec as
`Just (defaultStateCodec <n>)` where `<n>` is the hand-bumped `state-codec version=`
integer from the spec's snapshot block
(`keiro-dsl/src/Keiro/Dsl/Scaffold.hs:1697-1700`; spec syntax per
`keiro-dsl/test/fixtures/reservation-snapshot.keiro:38-39` — `snapshot every 100` /
`state-codec version=1 shape-hash="…"`).

Why this is the worst gap in the stack. Hydration (`keiro/src/Keiro/Command.hs:286-322`)
looks up a seed and, when the seeded replay of the *suffix* succeeds, never re-verifies the
seed against a full replay — the full-replay fallback at `Command.hs:306` runs only when
the seeded replay *fails*. A fold change (new `update` on one edge) does not make the
suffix replay fail; it just makes the seed wrong. Worse, after every successful append,
`verifyAndSnapshot` (`Command.hs:696-737`) replays only the just-appended batch *from the
possibly-stale pre-command state* and re-persists the result as a fresh snapshot at the new
head version (lines 723-737) — so the stale-derived state is laundered into a
current-looking snapshot and the full-replay escape hatch recedes with every command.
Minimal scenario: policy `Every 100`, snapshot at v200, deploy changes one edge's `update`
expression → same `(version, hash)` → the v200 seed plus 50 new-fold events produce a state
no replay would produce, which is re-snapshotted at v251, silently, forever.

Two aggravating facts. `keiro-dsl diff` has *zero* transition diffing — `aggregatePairDiff`
(`keiro-dsl/src/Keiro/Dsl/Diff.hs:351-356`) compares events, wire spec, and projections
only, and the word "transition" does not appear in `Diff.hs` at all. And the user docs are
actively wrong: `docs/user/snapshots.md:80-83` claims "When you change the manager's
register-file *or state shape*, the `shapeHash` changes automatically" (false for state
shape and for everything about the fold) and steers fold changes away from the manual
version bump.

Ownership boundaries from master plan 24 (Integration Points): this plan owns
`keiro/src/Keiro/Snapshot/Codec.hs`, the snapshot schema files, the keiki shape surface,
and — within `keiro-dsl` — only the fold-fingerprint lowering and the *transition-surface
diff advisory*. Plan 139 owns `keiro-dsl/src/Keiro/Dsl/Validate.hs` and the deprecation /
upcaster-chain diff rules; plan 140 owns `Scaffold.hs`'s upcaster lowering and
`Harness.hs`. This plan's `Scaffold.hs` edit is confined to `stateCodecExpr` and the new
fingerprint module it calls; its `Diff.hs` edit is confined to one new rule in
`aggregatePairDiff` plus one new `DiagnosticCode` constructor. keiki changes follow the
established upstream-in-scope precedent (master plan 9 made upstream keiki/kiroku work
in-scope for keiro plans when the fix belongs upstream).

Relevant ADRs: `docs/adr/` contains only
`0001-keiro-pgmq-job-processing-telemetry-contract.md`, which is unrelated to this plan. No
relevant ADR exists yet; this plan's completion creates the snapshot-discriminator ADR.

Term definitions. "Fold" — the pure function from (state, event) to state that replay
applies; in keiki it is the edge set's `update`/`target` structure. "Discriminator" — the
tuple of columns the snapshot lookup filters on; a seed is served only when all components
equal the current codec's values. "Control state" — the vertex type `s` of the transducer
(e.g. `ReservationVertex`), as opposed to the register file `rs`. "Spec-visible" — a change
expressible in the `.keiro` file itself (guards, writes, emits, states, register initials),
as opposed to a change made only in the hand-owned Holes module.

## Idempotence and Recovery

Every step is re-runnable. The migration uses `ADD COLUMN IF NOT EXISTS` with a constant
default, safe to re-apply and safe on tables of any size (Postgres adds a
non-volatile-default column without a table rewrite). Scaffolding overwrites only
`@generated`-bannered files; re-running it is a no-op when nothing changed. If the keiki
release stalls, keiro development continues against `cabal.project.local`; do not merge
keiro's use of the new keiki symbols until keiki 0.2.1.0 is actually published and the
registry serves it (then delete `cabal.project.local` and re-run the full bar). Rolling
back the keiro change on a migrated database is safe: old binaries ignore the extra column
entirely (their statements name columns explicitly), and the incompatibility disjunction in
the write statement lets a rolled-back codec replace newer rows, mirroring the existing
rollback affordance.
